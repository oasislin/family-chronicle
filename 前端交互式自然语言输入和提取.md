
这份设计的核心思路是将非结构化的自然语言，通过实体抽取（Extraction）、关系消歧（Disambiguation）与增量更新（Incremental Update）三个阶段，转化为结构化的家谱图谱，并以交互式对话实现数据闭环。

一、 系统逻辑架构：从自然语言到结构化程序
我们将整个过程拆解为三个核心模块，优先确保程序处理的严谨性：

解析层 (Parsing Layer)： 利用 NLP 提取语句中的人物实体、称谓词、时间及动作描述。

逻辑处理层 (Logic Processing Layer)： * 实体对齐： 判定“爷爷”是数据库中已有的“王大壮”，还是一个需要新增的节点。

拓扑更新： 根据称谓推导逻辑链条（如：A 是 B 的父亲，B 是 C 的父亲 -> A 是 C 的爷爷）。

对话反馈层 (Interaction Layer)： 将程序无法确定的“模糊项”以提问形式返还给用户。

二、 核心数据模型：回答关键三要素
程序在处理每一轮对话时，必须实时更新并回答以下三个维度：

2.1 人员识别（Who）
识别范围： 当前语句中出现的所有专有名词（姓名）或指代词（如“他”、“二叔”）。

2.2 关系链路（Relationship）
本句关系： 描述当前输入语句中，各个人员之间的直接连接（例如：A 带着 B 去买菜 -> A 与 B 存在同行或亲缘关系预判）。

全域关系： 将当前人员与数据库中已有的节点进行比对，建立长线连接（例如：当前句子的“小王”与库中已有的“老王”是父子关系）。

描述形式： 采用自然语言（如“王某是李某的长子”）进行存储与展示。

2.3 事件画像（Event）
提取内容： 识别语句中包含的时间、地点及动作（如：1998年全家搬迁至北京）。

价值： 为静态的家谱节点补充动态的“人物志”，形成结构化的家族大事记。

三、 人员匹配与确认逻辑 (Entity Resolution)
这是程序最关键的纠偏机制，分为三个层级执行：

精确匹配： 若姓名、年龄等特征与数据库中唯一人员完全吻合，自动挂载。

模糊猜测： * 根据上下文语境（如：提到“我爸”，而用户之前已定义过父亲身份）自动关联。

基于社交拓扑逻辑进行猜测（如：提到“二叔的儿子”，若库中已有二叔及他的孩子，则进行疑似匹配）。

用户闭环确认： * 当程序无法判定（出现同名、或全新身份）时，主动发起交互。

引导语示例： “你提到的‘老李’，是‘王大壮’的姐夫吗？”或“请问‘老李’和库里的哪位成员有关系？”

四、 交互式对话原型逻辑 (Interaction Workflow)
程序不再是冷冰冰的输入框，而是以“家族史记录官”的身份进行互动。

用户： “我记得我爷爷王大壮，在1950年的时候带着我爸爸王建国去了西安。”

程序处理：

提取： 王大壮（爷爷）、王建国（父亲）、1950年、西安（事件）。

匹配： 发现“王大壮”已在库中，“王建国”为新发现人员。

推理： 王大壮是王建国的父亲。

程序回复：
“明白了。我已经将王建国记录为王大壮的儿子。
此外，我记下了1950年他们一起前往西安的这段往事。

追问： 请问王建国先生还有其他的兄弟姐妹吗？” 这里面问题以确认为主，而不用顺势展开加入新的关系和人。

五、 交互式对话的数据考量
这里考虑两个核心点：
第一，现有数据存于本地文件中，需以最精简的方式将这些关系推送至大语言模型进行匹配或绑定。
第二，过去的历史不应仅以大模型的上下文窗口长度作为考量核心，这只能作为辅助；核心应是如何尽量压缩数据库中已有的关系，提供给大模型载入上下文，而不应依赖大模型的上下文进行记忆。
