# 关系推导引擎重构与数据架构设计 (Relationship Derivation Engine Redesign)

基于您的要求，本设计文档对现有关系推导引擎进行全面重构。目标是构建一个结构严谨、支持全网自动扩散、能处理复杂中文称谓、并具备良好交互体验（人机协同冲突解决）的底层架构。

## 1. 核心关系原则 (Core Relationship Principles)

关系网的数据底座必须足够简洁，以防止逻辑爆炸。我们将关系分为**底层直接关系（原子关系）**和**派生复合关系**。所有的中国式亲戚关系，在本质上都是原子关系的路径组合。

### 1.1 原子关系 (Atomic Relationships)
引擎的存储与基础推导仅依赖以下三种原子关系：
- `parent_child` (父母-子女)：需区分长幼（上下辈），天然自带性别属性。必须是唯一的生物学血缘关系。
- `spouse` (配偶)：同辈关系。支持多段婚姻（通过关系属性 `is_active=True/False` 区分现任与前任）。
- `sibling` (兄弟姐妹)：虽然可由 `parent_child` 派生，但考虑到有时不知道父母是谁，直接记录 `sibling` 也是必要的原子关系。

### 1.2 非血缘关系特例 (Special Non-Blood Relations)
必须通过明确的 subtype 或独立关系类型剥离出非血缘关系，避免污染血缘树：
- **继父母/继子 (Step-parents)**：单独作为 `step_parent_child` 关系，或者使用 `parent_child` 配合 `subtype="step"`。
- **过继/领养 (Adoption)**：建立 `parent_child` 连接，增加 `subtype="adopted"`。在法理/家族树上等同于血缘扩散，但严格逻辑上不具有生物学唯一性排他。
- **干亲 (Godparents)**：建立独立的 `godparent_godchild` 关系。不无限扩散血缘树，允许一个孩子有多个干爹干妈。

## 2. 称谓网络设计与消歧机制 (Appellation Network & Disambiguation)

由于中文称谓几乎无法穷举，且存在地域差异和口语表达不严谨的情况，我们需要在“预置词典”与“动态消歧”之间取得平衡。

### 2.1 高优预置词典与动态映射的结合
1. **冷启动与人工筛选的高优基准库 (Pre-seeded High-Confidence Dictionary)**：
   - 系统不会完全从零开始。我们会通过网络搜索或现有开源亲戚关系语料库，预生成一份基础称谓表。
   - **人工圈定**：我们通过人工干预，确定部分“高认可度、无歧义”的称谓（如：亲叔叔、外公、表哥等），并固化为其唯一的原子路径。
   - **低频词降级**：其余极低频、或地方方言专有、高度含混的词汇仅作为辅助参考，甚至从默认词典中删除，转交由 LLM 或人工消歧处理。
2. **LLM 实时解析辅导**：利用前端或后端接入的 LLM 的常识能力，将不在高优词典中的称谓动态翻译为“原子关系路径”。
3. **可动态更新的本地映射缓存 (Dynamic Title Dictionary)**：
   - 数据库维护一张动态的 `Appellation_Mapping` 表。
   - **支持一词多义（一对多映射）**：如果用户一开始定义“舅公”专指“母亲的舅舅”，之后又输入“父亲的舅舅也叫舅公”，系统会将这两条路径同时绑定到“舅公”这个称谓上。
   - 当遇到这种一对多的称谓时，它在后续输入中将自动被降级为“模糊称谓”，触发下方的消歧逻辑。

### 2.2 模糊称谓的提前消歧交互 (Interactive Disambiguation)
对于口语中不严谨的称谓（如“堂叔”、“表叔”）或在本地词典中**存在多条映射路径**的称谓：
- **拦截与提问**：引擎在建立站位人之前，必须强制触发**消歧询问 (Clarification Prompt)**。
- **交互示例**：
  - 用户：“新增人物B，他是A的舅公”。
  - 系统：发现“舅公”在当前词典中既有“父亲的舅舅”，也有“母亲的舅舅”。
  - 前端反问：“请问这位舅公是A爸爸那边的舅舅，还是妈妈那边的舅舅？”
- **结论**：在进入图数据库推导之前，必须通过多轮交互将路径**精确到唯一的原子关系链路**。

### 2.3 虚拟“站位人”机制 (Placeholder Nodes)
在消歧完成、明确了唯一路径后，若中间节点缺失，则使用站位人补齐逻辑链条：
- **动作**：例如路径为“爷爷 -> 爷爷的兄弟(站位人) -> 儿子(堂叔)”。系统静默创建无名站位人，设置属性 `is_placeholder = True`。
- **限制**：站位人不在前端列表展示，仅为维持图网络的连通性而存在。
- **最短路径原则 (Shortest Path Rule)**：系统在生成站位人时，强制计算从已知节点到目标节点的最短拓扑距离。**只要能用一个站位人讲通的逻辑，绝不用两个**。尽量压缩虚拟节点的数量，避免冗余网络膨胀。

## 3. 冲突检测与全网扩散 (Conflict Detection & Propagation)

**核心原则：绝不通过中文称谓进行冲突检测，而是将所有变更映射为原子关系图后，利用图论约束（Graph Constraints）进行严格的逻辑拦截。**

### 3.1 预设的底层图论冲突规则 (Graph Structural Constraints)

系统内置以下强制性的数据完整性与逻辑规则：

1. **生物学父母唯一性 (Biological Parent Uniqueness)**
   - 一个节点最多只能有一条来自 `Male` 节点的 `parent_child` 边和一条来自 `Female` 节点的边。

2. **现任配偶唯一性 (Active Spouse Uniqueness)**
   - 一个节点最多只能有一条 `is_active=True` 的 `spouse` 边。
   - **便捷更新体验**：在口语交流中，“发现他们早就离婚了”是非常高频的场景。因此，当触发双重配偶冲突时，系统必须提供低阻力的快捷更新入口。在前端弹窗中首选突出：“**将前任设为离异并更新为新配偶**”的选项卡，实现状态的一键平滑过渡。

3. **辈分有向无环图约束 (Generational DAG & Incest Prevention)**
   - 严禁下一代与同代产生同辈连接。一旦血系繁衍的树形图中出现辈分循环，系统立刻判定为严重冲突并报错拦截。

### 3.2 批量输入与必定触发的全网扩散更新 (System-Driven Propagation)

每次增加或修改节点关系时，都会执行全网 BFS 扩散。但在实际情况中，用户的一句话可能同时包含**多个新人物和多重关系**（如：“A的舅舅B娶了C，生了个儿子D”）。

- **操作队列化机制 (Operation Queuing)**：
  1. 系统首先将用户语句解析为一张“输入子图”（包含 A-B, B-C, B-D 的多条原子边）。
  2. 将这些原子边拆解为一个**按拓扑顺序排列的操作队列**（例如：先建B，再建C并连配偶，再建D并连父母）。
  3. 引擎**逐条 (Step-by-step)** 从队列中取出操作，执行写入并立即做 BFS 扩散与冲突检测。
  4. **中断与恢复**：如果队列执行到一半发生了冲突（比如发现C已经是别人的配偶），引擎将暂停整个队列，抛出排解交互给用户。用户排解完冲突后，引擎带着更新后的图谱，继续执行队列中剩余的操作。

### 3.3 人工介入的详细冲突排解流程示例 (Conflict Resolution Example)

* **场景 A：父母唯一性冲突**
  * 引擎输出：`{"type": "parent_uniqueness_violation", "node": "C", "existing_parent": "F1", "new_parent": "F2"}`
  * 前端提示：“C 的亲生父亲已经是 F1 了。对于新输入的 F2，请问：1. F2是继父/干爹；2. 之前的F1记录错了，F2才是亲生父亲；3. 取消。”

* **场景 B：配偶更新/冲突**
  * 前端提示：“A 目前的配偶是 B，现输入了配偶 C。请问：
    1. **【快捷更新】A 和 B 已经离婚/前任，现任是 C。**（一键降级前任，挂载新配偶）
    2. C 其实是前任配偶。
    3. 暂不记录。”

## 4. 交互层设计 (Interaction Layer)

引擎需为前端提供灵活的 API，支持两种主要的交互形态：

### 4.1 首选：自然语言持续问答 (Natural Language Continuous QA)
- 后端图引擎接收原子队列，放入沙盒推导。遇到阻碍时返回 `{"status": "needs_clarification", "type": "...", "options": [...]}`。
- LLM 将该状态转化为自然语言反问用户，用户确认后队列继续向下执行。

### 4.2 补充：UI 界面选取与模糊搜索 (UI Selection & Fuzzy Search)
在处理极其庞大的家族网时，提供基于图关系的智能检索：
1. **智能推荐 (Smart Recommendations)**：添加舅舅时，优先展示母亲那边的亲戚。
2. **关系模糊搜索 (Fuzzy Relational Search)**：搜索框不仅支持“名字”，还支持语义如“A母亲那边的亲戚”。
